# AirDB Cloud 产品定位与设计文档

## 1. 一句话定位

**AirDB Cloud 是面向非技术用户的云服务产品化平台**：把阿里云、腾讯云、AWS、Google Cloud 等主流云厂商的底层技术能力，重新封装成"开箱即用"的产品，让普通用户（个人站长、小微企业主、创业团队）无需具备云计算/运维知识，也能获得稳定、可靠、高效的云端服务。

我们不是云厂商，是云能力的"产品化二次封装层"——对用户屏蔽底层的账号体系、控制台、专业术语和运维复杂度，只暴露"我要一个什么服务"这个业务问题。

**对外表达方式上要克制**：产品要证明的是"这件事有多简单/多快搞定"，而不是罗列"我们支持多少种技术能力"。功能清单是内部规划语言，不是对用户说的话——这一点在第 4 章详细展开。

## 2. 目标用户画像

| 用户类型 | 痛点 | AirDB 提供的价值 |
|---|---|---|
| 个人站长/自媒体 | 想要个网站/博客，但看不懂云控制台里的 ECS、安全组、CDN 配置 | 几步选好套餐，自动上线，域名/证书/加速全包 |
| 小微企业主 | 需要官网、小程序后端、简单的数据存储，没有专职 IT | 按业务场景（"我要一个官网"、"我要存文件"）而不是按技术组件购买 |
| 早期创业团队 | 技术负责人精力有限，不想在基础设施上花太多心智 | 托管式运维、自动备份与告警，出问题有人兜底 |
| 独立开发者 | 想快速验证产品，不想在多个云厂商控制台之间切换 | 统一控制台、统一账单，底层可跨云调度 |
| AI 应用创业者 | 想做个 AI 应用/智能体，不想自己搭大模型网关、向量库、限流计费 | 场景化的 AI 能力套餐，接入即用 |
| 智能硬件/IoT 团队 | 设备要联网上报数据，不想自建 MQTT 网关和时序数据库 | 设备接入 + 数据存储 + 可视化一站式托管 |

共同特征：**懂业务，不懂/不想懂底层技术**；**为结果和确定性付费，而不是为资源单价谈判**。

## 3. 核心方法论：产品化，而非中转/代理

区别于简单的"云资源转售"，AirDB 的产品化体现在三层：

1. **场景化封装**：不卖"2核4G ECS"，卖"一个能扛住日常访问的网站托管套餐"。
2. **跨云抽象**：同一个产品背后，可能是阿里云、也可能是 AWS，用户不感知也不需要选择，AirDB 根据成本、地域、可用性动态调度。
3. **结果保障**：用户购买的是 SLA（在线率、备份、响应时长）承诺，而不是一堆需要自己拼装、自己兜底的组件。

## 4. 产品体验哲学：对标 Fly.io——做减法，而不是列清单

这是本轮修订新增、也是需要贯穿全局的一条原则。

### 4.1 Fly.io 做对了什么

- **不卖基础设施，卖"能跑起来"这件事**：用户体验到的是 `fly launch` 一条命令，背后的地域选择、VM 规格、网络配置全部被隐藏。
- **约定优于选择**：默认给出合理配置，只有真正需要深度定制的用户才会去碰高级选项。
- **全球部署是个"复选框"，不是架构决策**：多地域对用户来说只是打开一个开关，而不是要理解 CDN、Anycast、就近路由这些概念。
- **定价简单直白**：按资源用量清楚计价，不是需要拿计算器才能算明白的云厂商价目表。

### 4.2 对 AirDB 的启示

1. **官网/产品首页不做"能力罗列页"**：不摆"我们支持数据库、存储、AI、IoT、安全"这种功能条目列表，而是用一个具体场景直接演示——比如"从零到一个带数据库的网站上线，用时 3 步 / 60 秒"。
2. **一个动作贯穿所有产品线**：无论是建站、数据库还是 AI 能力，用户的操作路径都应该是同构的"选场景 → 一键开通 → 查看健康度"，而不是每条产品线各自一套复杂控制台。
3. **默认无需选择底层**：地域、云厂商、规格都是系统决策，用户想深度定制时才暴露"高级选项"。
4. **第 5 章"核心产品矩阵"的定位需要重新理解**：那张表是**内部能力清单/规划工具**，用来指导我们做什么、接入哪些底层能力，**不是对外的营销素材**。对外叙事要靠体验演示，不靠功能列表。

### 4.3 一个关键差异：Fly.io 自己拥有基础设施，AirDB 不是

需要提醒的一点：Fly.io 的简单，很大一部分建立在它**自建全球机房、自研 Firecracker microVM**之上——它是基础设施的直接拥有者，简单是"全栈自控"的结果。

AirDB 的模式更接近 **Railway、Render** 这类"在别人的云（AWS/GCP等）之上做体验层"的打法——难度更大，因为简单感是在别人的复杂系统上"抽象"出来的，不是自己天然拥有的。这意味着：

- 对内，跨云调度、故障切换的工程复杂度会比 Fly.io 更高（因为要适配多个厂商的差异化 API 和行为）；
- 对外，仍然可以做到同等甚至更简单的体验，但需要更强的中间层工程能力，这一点建议写进技术架构文档，而不只是定位文档。

## 5. 核心产品矩阵（内部能力清单，非对外话术）

围绕"用户的业务问题"划分产品线，而不是围绕"云厂商的服务名称"划分：

### 5.1 建站与应用托管
- 网站/H5/小程序后端一键部署
- 自动 HTTPS 证书、自动扩缩容
- 底层能力：ECS/EC2、容器服务、CDN、负载均衡

### 5.2 数据库与数据服务
- 托管数据库（MySQL/PostgreSQL/Redis 等），一键创建、自动备份、慢查询与容量告警
- 数据迁移与同步工具（老系统搬家）
- 底层能力：RDS、云数据库、DTS

### 5.3 存储与备份
- 文件/图片/视频云存储，自动 CDN 加速分发
- 定期自动备份 + 一键恢复（面向非技术用户的"后悔药"）
- 底层能力：对象存储（OSS/S3/GCS）、备份快照

### 5.4 域名、网络与安全
- 域名注册/解析、SSL 证书自动续期
- DDoS/CC 防护、WAF 基础防护
- 底层能力：DNS、CDN、云防火墙

### 5.5 监控与托管运维（隐形但核心）
- 7x24 自动监控 + 告警 + 人工兜底响应
- 用户看到的是"服务健康度"仪表盘，而不是原始监控指标
- 这一层是让用户"敢把业务交出来"的信任基础，也是后续可扩展的增值服务（例如按 SLA 等级分层收费）

### 5.6 AI 能力（新增）
- 大模型/推理网关：封装主流大模型 API，用户买的是"智能客服"、"内容生成"这类场景化能力，而不是自己对接模型厂商
- 向量数据库/RAG 托管：与 5.2 数据库能力打通，做到"上传文档即可问答"
- 定位为**能力层**，不作为独立产品线大肆宣传，而是嵌入建站/数据库产品，作为"加个开关"式的增值能力（呼应第 4 章的体验哲学）

### 5.7 IoT 设备接入（新增）
- 设备接入网关（MQTT 等协议托管），免去用户自建协议网关
- 时序数据存储与可视化，复用 5.2/5.5 的数据与监控能力
- 目标用户相对垂直（智能硬件团队），可作为第二阶段拓展方向，不建议第一阶段就铺开

> 优先级建议：**建站与应用托管、数据库与数据服务** 作为第一阶段主打产品（用户感知最强、决策链最短）；存储/域名/安全作为随选配套；AI 能力作为数据库/建站产品的"增值开关"尽早引入（时代红利期，用户预期值高）；IoT 作为第二阶段独立评估；监控运维贯穿所有产品，是差异化竞争力所在，不单独出售，而是嵌入到每个产品的"稳定可靠"承诺里。

## 6. 服务保障体系（核心竞争力）

这是 AirDB 相对于用户"直接注册阿里云/AWS 自己搭"的核心溢价点，必须具体、可验证、可对外承诺：

| 维度 | 承诺方向（示例，需结合实际底层 SLA 细化） |
|---|---|
| **可用性** | 服务在线率承诺（如 99.9%），不达标有补偿机制 |
| **数据可靠性** | 多副本存储 + 定期自动备份 + 明确的备份保留周期与恢复时限（RPO/RTO） |
| **安全合规** | 传输与存储加密、访问权限最小化、符合国内等保/国际主流合规要求（视目标市场而定） |
| **容灾能力** | 关键服务多地域/多可用区部署，单点故障自动切换 |
| **响应时效** | 明确的工单/告警响应 SLA（如 30 分钟响应、4 小时内解决高优先级问题） |
| **透明度** | 对外公开的服务状态页（Status Page），历史故障与处理记录可查 |

这一节在正式对外文档中需要转化为**可以写进服务协议的具体数字**，建议下一步和你确认每个产品线的 SLA 目标值。

## 7. 差异化对比

### 7.1 vs 直接使用阿里云/AWS

| 维度 | 直接使用云厂商 | AirDB Cloud |
|---|---|---|
| 购买单元 | 技术组件（实例规格、存储桶策略等） | 业务场景（"网站托管"、"数据库套餐"） |
| 学习成本 | 需要理解云厂商专业术语与控制台 | 无需了解底层，按需求选择 |
| 跨云能力 | 需自己对接多个厂商账号与账单 | 统一入口、统一账单，底层可跨云调度 |
| 运维责任 | 用户自行负责监控、备份、告警处理 | AirDB 托管，用户只对接"结果" |
| 出问题时 | 自己排查、自己联系云厂商工单 | AirDB 作为统一责任方兜底 |

### 7.2 vs Fly.io / Railway / Render 这类平台

| 维度 | Fly.io / Railway / Render | AirDB Cloud |
|---|---|---|
| 用户群体 | 以开发者为主，CLI/Git 驱动 | 开发者 + 完全不懂技术的普通用户 |
| 覆盖范围 | 聚焦应用托管 | 应用托管 + 数据库 + 存储 + AI + IoT 等更宽的场景 |
| 基础设施归属 | Fly.io 自建；Railway/Render 基于云厂商 | 基于阿里云/腾讯云/AWS/GCP 等多云 |
| 市场重心 | 海外为主 | 国内 + 海外并重（可差异化的空间） |

## 8. 商业模式初步方向

- **套餐订阅制为主**：按场景打包（如"个人建站套餐"、"小微企业套餐"），价格透明、无需理解底层计费单位，呼应第 4 章"简单直白定价"的原则
- **用量超额按量计费为辅**：套餐内额度用完后按实际用量补差价，避免用户对"按量计费"产生恐惧感
- **AI/IoT 等能力作为套餐内加购开关**，而非独立复杂定价体系
- 具体价格点需要结合底层云厂商成本核算，本文档暂不展开，建议单独出一份定价模型文档

## 9. 风险与挑战

- **强依赖底层云厂商 API**：接口变更、限流、区域下线等都会直接影响 AirDB 的稳定性，需要有适配层和降级预案
- **多云调度的工程复杂度**：不同厂商的网络、计费、故障行为差异大，"用户无感知跨云"在工程上比 Fly.io 的自建模式更难实现，第一阶段建议只做单云或双云，不追求完全智能调度
- **SLA 兜底的成本压力**：承诺的可用性/响应时效如果高于底层云厂商本身的 SLA，差额风险和补偿成本由 AirDB 承担，需要精算
- **"简单"与"能力广度"的张力**：AI、IoT 等新能力越加越多，越容易滑回"功能罗列"的老路，需要有意识地克制，用体验设计而不是功能堆砌来体现价值

## 10. MVP 建议范围

第一阶段建议聚焦、不铺开：

1. **1个拳头产品**：建站托管 或 数据库托管（二选一优先跑通）
2. **底层先接入 1-2 家云厂商**（如阿里云 + 一家海外云，覆盖国内外场景），跨云调度能力后续再做
3. **服务保障做"看得见"**：哪怕初期人工介入较多，也要有状态页、有备份策略、有明确响应时间承诺
4. 控制台/官网只需呈现"选套餐 → 开通 → 查看健康度"三步，隐藏所有底层配置项
5. AI 能力可作为 MVP 的一个"加分开关"尽早引入（例如数据库产品自带"开启 AI 问答"选项），但不单独作为拳头产品
6. IoT 暂不纳入 MVP，作为第二阶段方向观察

## 11. 待你确认的开放问题

- 第一阶段主打产品：建站托管 vs 数据库托管，还是两者并行？
- 目标市场：国内为主（对接阿里云/腾讯云），还是国内外并重（需要接入 AWS/GCP）？
- 服务保障的具体 SLA 数字（可用性%、备份周期、响应时长），是否已有初步目标？
- 商业模式：订阅制套餐、按用量计费，还是两者结合？
- AI 能力的深度：只做"加个开关"式的场景化能力，还是要做成独立主打产品？
- 是否认可"官网不做功能罗列页，改用场景演示"这个方向？这会直接影响后续官网/落地页的设计方式。
