# AirDB Cloud 产品定位与设计文档

## 1. 一句话定位

**AirDB Cloud 是面向非技术用户的云服务产品化平台**：把阿里云、腾讯云、AWS、Google Cloud 等主流云厂商的底层技术能力，重新封装成"开箱即用"的产品，让普通用户（个人站长、小微企业主、创业团队）无需具备云计算/运维知识，也能获得稳定、可靠、高效的云端服务。

我们不是云厂商，是云能力的"产品化二次封装层"——对用户屏蔽底层的账号体系、控制台、专业术语和运维复杂度，只暴露"我要一个什么服务"这个业务问题。

## 2. 目标用户画像

| 用户类型 | 痛点 | AirDB 提供的价值 |
|---|---|---|
| 个人站长/自媒体 | 想要个网站/博客，但看不懂云控制台里的 ECS、安全组、CDN 配置 | 几步选好套餐，自动上线，域名/证书/加速全包 |
| 小微企业主 | 需要官网、小程序后端、简单的数据存储，没有专职 IT | 按业务场景（"我要一个官网"、"我要存文件"）而不是按技术组件购买 |
| 早期创业团队 | 技术负责人精力有限，不想在基础设施上花太多心智 | 托管式运维、自动备份与告警，出问题有人兜底 |
| 独立开发者 | 想快速验证产品，不想在多个云厂商控制台之间切换 | 统一控制台、统一账单，底层可跨云调度 |

共同特征：**懂业务，不懂/不想懂底层技术**；**为结果和确定性付费，而不是为资源单价谈判**。

## 3. 核心方法论：产品化，而非中转/代理

区别于简单的"云资源转售"，AirDB 的产品化体现在三层：

1. **场景化封装**：不卖"2核4G ECS"，卖"一个能扛住日常访问的网站托管套餐"。
2. **跨云抽象**：同一个产品背后，可能是阿里云、也可能是 AWS，用户不感知也不需要选择，AirDB 根据成本、地域、可用性动态调度。
3. **结果保障**：用户购买的是 SLA（在线率、备份、响应时长）承诺，而不是一堆需要自己拼装、自己兜底的组件。

## 4. 核心产品矩阵

围绕"用户的业务问题"划分产品线，而不是围绕"云厂商的服务名称"划分：

### 4.1 建站与应用托管
- 网站/H5/小程序后端一键部署
- 自动 HTTPS 证书、自动扩缩容
- 底层能力：ECS/EC2、容器服务、CDN、负载均衡

### 4.2 数据库与数据服务
- 托管数据库（MySQL/PostgreSQL/Redis 等），一键创建、自动备份、慢查询与容量告警
- 数据迁移与同步工具（老系统搬家）
- 底层能力：RDS、云数据库、DTS

### 4.3 存储与备份
- 文件/图片/视频云存储，自动 CDN 加速分发
- 定期自动备份 + 一键恢复（面向非技术用户的"后悔药"）
- 底层能力：对象存储（OSS/S3/GCS）、备份快照

### 4.4 域名、网络与安全
- 域名注册/解析、SSL 证书自动续期
- DDoS/CC 防护、WAF 基础防护
- 底层能力：DNS、CDN、云防火墙

### 4.5 监控与托管运维（隐形但核心）
- 7x24 自动监控 + 告警 + 人工兜底响应
- 用户看到的是"服务健康度"仪表盘，而不是原始监控指标
- 这一层是让用户"敢把业务交出来"的信任基础，也是后续可扩展的增值服务（例如按 SLA 等级分层收费）

> 优先级建议：**建站与应用托管、数据库与数据服务** 作为第一阶段主打产品（用户感知最强、决策链最短）；存储/域名/安全作为随选配套；监控运维贯穿所有产品，是差异化竞争力所在，不单独出售,而是嵌入到每个产品的"稳定可靠"承诺里。

## 5. 服务保障体系（核心竞争力）

这是 AirDB 相对于用户"直接注册阿里云/AWS 自己搭"的核心溢价点，必须具体、可验证、可对外承诺：

| 维度 | 承诺方向（示例，需结合实际底层 SLA 细化） |
|---|---|
| **可用性** | 服务在线率承诺（如 99.9%），不达标有补偿机制 |
| **数据可靠性** | 多副本存储 + 定期自动备份 + 明确的备份保留周期与恢复时限（RPO/RTO） |
| **安全合规** | 传输与存储加密、访问权限最小化、符合国内等保/国际主流合规要求（视目标市场而定） |
| **容灾能力** | 关键服务多地域/多可用区部署，单点故障自动切换 |
| **响应时效** | 明确的工单/告警响应 SLA（如 30 分钟响应、4 小时内解决高优先级问题） |
| **透明度** | 对外公开的服务状态页（Status Page），历史故障与处理记录可查 |

这一节在正式对外文档中需要转化为**可以写进服务协议的具体数字**，建议下一步和你确认每个产品线的 SLA 目标值。

## 6. 与"直接用阿里云/AWS"的差异化

| 维度 | 直接使用云厂商 | AirDB Cloud |
|---|---|---|
| 购买单元 | 技术组件（实例规格、存储桶策略等） | 业务场景（"网站托管"、"数据库套餐"） |
| 学习成本 | 需要理解云厂商专业术语与控制台 | 无需了解底层，按需求选择 |
| 跨云能力 | 需自己对接多个厂商账号与账单 | 统一入口、统一账单，底层可跨云调度 |
| 运维责任 | 用户自行负责监控、备份、告警处理 | AirDB 托管，用户只对接"结果" |
| 出问题时 | 自己排查、自己联系云厂商工单 | AirDB 作为统一责任方兜底 |

## 7. MVP 建议范围

第一阶段建议聚焦、不铺开：

1. **1个拳头产品**：建站托管 或 数据库托管（二选一优先跑通）
2. **底层先接入 1-2 家云厂商**（如阿里云 + 一家海外云，覆盖国内外场景），跨云调度能力后续再做
3. **服务保障做"看得见"**：哪怕初期人工介入较多，也要有状态页、有备份策略、有明确响应时间承诺
4. 控制台/官网只需呈现"选套餐 → 开通 → 查看健康度"三步，隐藏所有底层配置项

## 8. 待你确认的开放问题

- 第一阶段主打产品：建站托管 vs 数据库托管，还是两者并行？
- 目标市场：国内为主（对接阿里云/腾讯云），还是国内外并重（需要接入 AWS/GCP）？
- 服务保障的具体 SLA 数字（可用性%、备份周期、响应时长），是否已有初步目标？
- 商业模式：订阅制套餐、按用量计费，还是两者结合？
